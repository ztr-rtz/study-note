## 一、网络基础

### 1. OSI七层模型（理论模型）

从上到下：

| 层次 | 功能 | 典型设备/协议 |
|------|------|---------------|
| 应用层 | 文件传输、邮件等 | HTTP, FTP, DNS |
| 表示层 | 加解密、压缩解压 | - |
| 会话层 | 建立数据传输通道 | - |
| 传输层 | 传输方式、端口号 | TCP, UDP |
| 网络层 | 数据路由 | 路由器, IP |
| 数据链路层 | 封装成帧、点对点通信 | 交换机, ARP |
| 物理层 | 物理设备标准 | 网线, 光纤 |

### 2. TCP/IP模型（工程实际用）

四层模型，把OSI的上三层合并、下两层合并：

| TCP/IP层 | 对应OSI层 | 典型协议 |
|-----------|-----------|----------|
| 应用层 | 应用+表示+会话 | HTTP, HTTPS, FTP, TFTP, MQTT, DNS |
| 传输层 | 传输层 | TCP, UDP |
| 网络层 | 网络层 | IP（IPv4/IPv6） |
| 网络接口层 | 数据链路+物理 | ARP, RARP |

也有五层模型的说法，就是把网络接口层拆开。考试/面试一般问四层。

### 3. IP协议

**IPv4**：32位，点分十进制表示，如 `192.168.0.103`

IP地址 = 网络位 + 主机位，靠子网掩码区分：
- 子网掩码为1的位 → 网络位（标识网段）
- 子网掩码为0的位 → 主机位（标识网段内第几台主机）

例：`192.168.0.103` + 子网掩码 `255.255.255.0` → 网段号 `192.168.0.0`

**特殊IP地址**：

| 类型 | 说明 | 举例 |
|------|------|------|
| 网段号 | 网络位不变，主机位全0 | 192.168.1.0 |
| 广播号 | 网络位不变，主机位全1 | 192.168.1.255 |
| 回环地址 | 本机自测 | 127.0.0.0 |

**IP地址分类**：

| 类别 | 范围 | 子网掩码 | 私有IP范围 |
|------|------|----------|------------|
| A类 | 1.0.0.0 ~ 126.255.255.255 | 255.0.0.0 | 10.0.0.0 ~ 10.255.255.255 |
| B类 | 128.0.0.0 ~ 191.255.255.255 | 255.255.0.0 | 172.16.0.0 ~ 172.31.255.255 |
| C类 | 192.0.0.0 ~ 223.255.255.255 | 255.255.255.0 | 192.168.0.0 ~ 192.168.255.255 |
| D类 | 224.0.0.0 ~ 239.255.255.255 | - | 组播/广播 |
| E类 | 240.0.0.0 ~ 255.255.255.254 | - | 实验用 |

公有IP：电信分配，付费，可直接访问Internet
私有IP：不能直接访问Internet

### 4. 端口号

区分同一主机上的不同网络进程，16位，范围 0~65535。

| 范围 | 用途 | 常见端口 |
|------|------|----------|
| 1~1023 | 系统服务 | HTTP:80, HTTPS:443, FTP:20/21, TFTP:69, MQTT:1883 |
| 1024~49151 | 注册端口 | IANA指定 |
| 49152~65535 | 动态/私有 | 临时使用 |

### 5. 网络配置命令

```bash
ifconfig          # 查看IP地址信息
ping IP/域名       # 检查网络连通性
```

## 二、UDP编程

### 1. C/S vs B/S模型

| 模型 | 客户端 | 特点 |
|------|--------|------|
| C/S | 专用客户端 | 客户端和服务端都要开发，双方都可保存资源 |
| B/S | 通用浏览器 | 只维护服务端，用HTTP协议，资源集中在服务端 |

### 2. UDP特点

- 面向数据包
- 不需要建立连接
- 尽最大可能交付，不安全不可靠
- 机制简单，实时性高，效率高

### 3. UDP编程核心函数

**创建套接字**：

```c
int socket(int domain, int type, int protocol);
// domain: AF_INET(IPv4) / AF_INET6(IPv6)
// type:   SOCK_STREAM(TCP) / SOCK_DGRAM(UDP)
// 返回: 成功文件描述符, 失败-1
```

套接字本质就是文件描述符，系统给应用层提供的网络通信端口。

**发送数据**：

```c
ssize_t sendto(int sockfd, const void *buf, size_t len, int flags,
               const struct sockaddr *dest_addr, socklen_t addrlen);
// dest_addr: 接收方地址信息
// 返回: 实际发送字节数, 失败-1
```

**接收数据**：

```c
ssize_t recvfrom(int sockfd, void *buf, size_t len, int flags,
                 struct sockaddr *src_addr, socklen_t *addrlen);
// src_addr: 保存发送方地址
// 返回: 实际收到字节数, 失败-1
```

**绑定地址**：

```c
int bind(int sockfd, const struct sockaddr *addr, socklen_t addrlen);
// 把套接字和本机IP、端口号绑定
// 返回: 成功0, 失败-1
```

### 4. 地址结构体

```c
struct sockaddr_in {
    sa_family_t    sin_family;  // AF_INET
    in_port_t      sin_port;    // 端口，网络字节序
    struct in_addr sin_addr;    // IP地址，网络字节序
};

struct in_addr {
    uint32_t       s_addr;
};
```

### 5. 字节序转换

主机字节序是小端，网络字节序是大端，必须转换：

| 函数 | 作用 | 说明 |
|------|------|------|
| `htonl(hostlong)` | 主机→网络 | 32位 |
| `htons(hostshort)` | 主机→网络 | 16位 |
| `ntohl(netlong)` | 网络→主机 | 32位 |
| `ntohs(netshort)` | 网络→主机 | 16位 |

记忆：h=host, n=net, l=long(32位), s=short(16位)

### 6. IP地址格式转换

```c
in_addr_t inet_addr(const char *cp);      // 字符串IP → 整形IP
char *inet_ntoa(struct in_addr in);        // 整形IP → 字符串IP
```

### 7. UDP头部（8字节）

| 字段 | 大小 |
|------|------|
| 源端口号 | 2字节 |
| 目标端口号 | 2字节 |
| 长度 | 2字节 |
| 校验和 | 2字节 |

### 8. UDP编程流程

服务端：`socket` → `bind` → `recvfrom` → `sendto` → `close`
客户端：`socket` → `sendto` → `recvfrom` → `close`

## 三、TCP编程

### 1. TCP特点

- 面向数据流
- 面向连接
- 安全可靠
- 机制复杂

### 2. 三次握手与四次挥手

**三次握手**（建立连接）：双方确认都准备好了再通信

**四次挥手**（断开连接）：确保双方都收发完毕再断开，客户端或服务端都可以发起

### 3. TCP编程核心函数

**请求连接（客户端）**：

```c
int connect(int sockfd, const struct sockaddr *addr, socklen_t addrlen);
// addr: 服务端地址
// 返回: 成功0, 失败-1
```

**监听（服务端）**：

```c
int listen(int sockfd, int backlog);
// backlog: 允许同时监听的最大客户端数
// 返回: 成功0, 失败-1
```

**接收客户端连接（服务端）**：

```c
int accept(int sockfd, struct sockaddr *addr, socklen_t *addrlen);
// 返回: 成功返回通讯套接字(新fd), 失败-1
```

注意：accept返回的是新的通讯套接字，原来的监听套接字继续监听。

**接收数据**：

```c
ssize_t recv(int sockfd, void *buf, size_t len, int flags);
// 返回: 实际收到字节数; 对方关闭返回0; 失败-1
```

### 4. TCP编程流程

服务端：`socket` → `bind` → `listen` → `accept` → `recv`/`send` → `close`
客户端：`socket` → `connect` → `send`/`recv` → `close`

### 5. TCP粘包问题

**现象**：接收端收到多包数据粘连在一起

**原因**：
1. 收发速率不匹配，接收缓冲区缓存了多包数据
2. 发送端把多包数据重新组包发送

**解决方案**：

| 方案 | 做法 | 注意 |
|------|------|------|
| 收发指定大小 | 用结构体固定大小 | 不同平台结构体字节数可能不同 |
| 加分隔符 | 每包末尾加分隔符，如 `\n` | 简单但不够严谨 |
| 自定义协议 | 帧头+数据长度+数据+校验和+帧尾 | 最可靠 |

常用帧头帧尾：`7E`、`5A A5`、`EB90`
常用校验：8位和校验、16位和校验、CRC校验

### 6. TCP可靠性机制

**安全可靠相关**：

| 机制 | 说明 |
|------|------|
| 应答机制 | 每包数据都有ACK应答，序列号+确认号 |
| 超时重传 | 等待ACK超时则重发 |
| 流量控制 | 根据响应报文窗口大小动态调整发送速率 |
| 滑动窗口 | 缓存已发送已应答、已发送未应答、待发送数据 |

**提高效率相关**：

| 机制 | 说明 |
|------|------|
| 延迟应答 | 发送数据同时等待应答 |
| 捎带应答 | ACK可以和正文数据一起发出 |

## 四、HTTP协议

### 1. URL

统一资源定位符，格式：

```
协议://域名:端口/路径?参数&参数
```

参数是键值对形式：`关键字=值`

HTTP默认端口80，HTTPS默认端口443

### 2. HTTP协议

应用层协议，基于TCP。

**请求报文格式**：

```
GET / HTTP/1.1\r\n
Host: news.sohu.com\r\n
User-Agent: Mozilla/5.0 ...\r\n
Accept: text/html,...\r\n
\r\n
```

**响应报文格式**：

```
HTTP/1.1 200 OK\r\n
Date: Wed, 17 Jun 2026 ...\r\n
Content-Type: text/html;charset=utf-8\r\n
Connection: keep-alive\r\n
\r\n
<!DOCTYPE html>...
```

**长连接 vs 短连接**：
- `Connection: keep-alive` → 长连接，服务端保持连接一段时间
- `Connection: close` → 短连接，服务端立即断开

## 五、TCP并发服务器

| 方式 | 特点 |
|------|------|
| 多进程 | 安全性高，资源开销大，并发量小 |
| 多线程 | 开销比进程小，并发量比进程大 |
| 线程池 | 提前创建大量线程管理，避免反复创建销毁 |
| IO多路复用 | 单进程监测多个fd，效率最高 |

线程池用到生产者-消费者设计模式。

## 六、Linux常用IO模型

### 四种IO模型对比

| 模型 | 特点 | 缺点 |
|------|------|------|
| 阻塞IO | fgets/read/fread/recv，让IO有先后顺序，省CPU | 单线程只能等一个IO |
| 非阻塞IO | 搭配轮询使用 | 浪费CPU |
| 信号驱动IO | 效率高 | 信号个数有限，无法区分多个IO事件 |
| IO多路复用 | 单进程同时监测多个IO | 需要掌握select/poll/epoll |

### select / poll / epoll 对比

| 对比项 | select | poll | epoll |
|--------|--------|------|-------|
| 集合存储 | 数组(位图) | 链表 | 红黑树 |
| fd上限 | 1024 | 理论无限制 | 理论无限制 |
| 集合位置 | 应用层 | 应用层 | 内核层 |
| 数据拷贝 | 应用层↔内核层反复拷贝 | 同左 | 直接在内核层操作 |
| 查找事件 | 遍历 | 遍历 | 直接返回到达事件 |
| 触发模式 | 仅水平触发(低速) | 仅水平触发(低速) | 水平+边沿(高速) |

### epoll核心函数

```c
int epoll_create(int size);
// 创建fd集合，返回集合代表的fd

int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);
// op: EPOLL_CTL_ADD / EPOLL_CTL_MOD / EPOLL_CTL_DEL

int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);
// timeout: -1不设超时
// 返回: 到达事件个数
```

事件结构体：

```c
struct epoll_event {
    uint32_t      events;   // 事件类型
    epoll_data_t  data;     // 用户数据（可存fd）
};
```

## 八、易错提醒

1. **bind的地址类型转换**：bind参数是 `struct sockaddr*`，但实际填的是 `struct sockaddr_in`，需要强制类型转换 `(struct sockaddr*)&addr`

2. **端口号必须转网络字节序**：`addr.sin_port = htons(8080);` 忘了htons会导致连不上

3. **accept返回新fd**：accept返回的是通讯套接字，不是原来的监听套接字。监听套接字继续用来accept

4. **recv返回0的含义**：不是出错，是对方关闭了连接。返回-1才是出错

5. **UDP不需要connect/listen/accept**：UDP是无连接的，直接sendto/recvfrom

6. **TCP粘包不是bug**：是TCP流式传输的正常现象，需要应用层自己处理分包

7. **epoll的fd集合在内核层**：select/poll的集合在应用层，每次调用都要拷贝到内核，这是epoll效率高的关键原因

8. **信号函数要短**：signal注册的处理函数尽量简短，避免阻塞和耗时操作。推荐用sigaction替代signal，可以屏蔽同源信号

9. **SIGKILL和SIGSTOP不能被捕获或忽略**：这是系统强制的，没有办法绕过

10. **网络字节序是大端**：主机是小端，网络是大端，数据发到网络前必须转
