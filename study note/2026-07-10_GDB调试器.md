# GDB 调试器

> Linux 软件开发四大必备工具：gcc（编译）、vim（编辑）、gdb（调试）、makefile（工程管理）

---

## 一、常用命令速查表

| 命令 | 全称 | 作用 | 示例 |
|------|------|------|------|
| `r` | run | 运行程序，可跟命令行参数 | `r` / `r 12 3` |
| `where` / `bt` | — | 查看栈结构（函数调用链），逆序输出 | `where` / `bt` |
| `b` | break | 设置断点（行号/函数名/文件:行号） | `b 14` / `b fun` / `b 1.c:16` |
| `n` | next | 单步执行，函数调用直接跳过（不进入） | `n` |
| `s` | step | 单步执行，遇到自定义函数进入函数体 | `s` |
| `p` | print | 打印变量/指针内容 | `p a` / `p *a` |
| `display` | — | 持续显示变量值（每次停断点都刷新） | `display a` |
| `c` | continue | 运行到下一个断点，常用于跳出循环 | `c` |
| `return` | — | 直接跳出当前函数，回到调用点继续 | `return` |
| `l` | list | 显示源代码，默认从 main 所在文件开始，每次 10 行 | `l` / `l 10` |
| `q` | quit | 退出 GDB | `q` |

## 二、一般调试流程

**核心思路：设断点 → 运行 → 看变量 → 单步推进 → 循环直到定位错误**

1. **编译加 `-g`**
   ```bash
   gcc -g 1.c          # 给 a.out 加入调试信息，否则 GDB 无法关联源码
   ```

2. **启动 GDB**
   ```bash
   gdb ./a.out
   ```

3. **设置断点**（三种方式）
   - `b 16` — 在 main 函数第 16 行
   - `b 1.c:36` — 在 1.c 文件第 36 行（多文件场景）
   - `b fun` — 在 fun 函数入口（任何调用触发暂停）

4. **运行程序**
   - `r` — 无参数运行
   - `r 1 2` — 带命令行参数运行（等同于 `./a.out 1 2`）

5. **在断点处检查**
   - `p 变量` — 看值
   - `p *指针` — 看指针所指内容

6. **单步推进**
   - 当前行有函数调用 → `n` 跳过整个函数 / `s` 进入函数体（只进自定义函数）
   - 无函数调用 → `n` 和 `s` 效果一样

7. **重复 5~6 步**，直到找到逻辑错误

## 三、段错误调试流程

**段错误（Segmentation Fault）不需要提前设断点，靠栈回溯定位崩溃点**

1. **编译加 `-g`**
   ```bash
   gcc -g 1.c
   ```

2. **启动 GDB**
   ```bash
   gdb ./a.out
   ```

3. **直接运行**（不设断点）
   - `r` 或 `r 1 2`（带参数时）

4. **重现错误** — 输入必要数据，等待段错误触发

5. **查栈定位**
   - 输入 `where` 或 `bt`
   - 栈信息会显示崩溃发生的源文件和行号（如 `linklinst.c:71`）
   - 根据行号分析错误原因（非法指针访问、越界等）

## 四、TUI 模式

```bash
layout src      # 启动 GDB 时直接进入源码可视化模式
```

效果：在 GDB 终端内分屏显示源代码，断点和当前行高亮，更直观地跟踪执行位置。

## 五、易错提醒

| 陷阱 | 说明 |
|------|------|
| 忘记 `-g` | 编译时不加 `-g`，GDB 无法显示源码和行号，调试基本失效 |
| `s` 进库函数 | `s` 只进入**自定义函数**，不会进入库函数内部 |
| 断点设错文件 | 多文件项目务必用 `b 文件名:行号` 格式，避免断到错误文件 |
| 段错误没输入 `bt` | 段错误发生后一定要用 `where/bt` 看栈，否则无从定位 |
| `display` 忘取消 | `display` 持续输出会干扰视线，用 `undisplay 编号` 关掉 |
