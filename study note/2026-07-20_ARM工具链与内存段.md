# ARM 交叉工具链与内存段

## 一、ARM 工具链：gcc / ld / objcopy / objdump

### 为什么需要交叉工具链

开发主机是 x86 Ubuntu，目标板是 ARM（i.MX6ULL / Cortex-A7）。x86 的 gcc 编出来的机器码跑在 ARM 上跑不了，必须用 ARM 交叉编译器：`arm-linux-gnueabihf-gcc`。

```
x86 Ubuntu (开发主机)          ARM i.MX6ULL (目标板)
  gcc → x86 机器码               只能执行 ARM 机器码
  arm-linux-gnueabihf-gcc → ARM 机器码 ✓
```

Vscode 写代码，Ubuntu 编译，生成的 .bin 烧到板子上跑。

### 四个核心工具

整个从源码到可烧录文件的流程：

```
*.c / *.S  →  gcc  →  *.o  →  ld  →  *.elf  →  objcopy  →  *.bin
                                    ↘ objdump → *.dis (反汇编，调试用)
```

#### 1. gcc — 编译器

干的活：预处理 → 编译 → 汇编，一步到位把 `.c`/`.S` 变成 `.o` 目标文件。

```bash
# 编译汇编文件
arm-linux-gnueabihf-gcc -c led.S -o led.o

# 编译C文件
arm-linux-gnueabihf-gcc -c main.c -o main.o
```

`-c` 表示只编译不链接，输出 `.o`。如果不加 `-c`，gcc 会自动调 ld 完成链接直接生成可执行文件——但裸机开发一般要手动调 ld 用链接脚本控制内存布局，所以分开做。

编译流程四步（gcc 内部自动完成）：
```
*.c → 预处理(展开宏/包含头文件) → *.i → 编译(生成汇编) → *.s → 汇编(生成机器码) → *.o
```

#### 2. ld — 链接器

干的活：把多个 `.o` 文件拼成一个 `.elf` 可执行文件，按照链接脚本（`.lds`）指定各段在内存中的位置。

```bash
arm-linux-gnueabihf-ld -Timx6ull.lds led.o main.o -o led.elf
```

链接脚本长这样，决定代码和数据放到哪个地址：
```
SECTIONS
{
    . = 0x87800000;          // 链接起始地址（i.MX6ULL DDR 起始）
    .text : { *(.text*) }    // 代码段
    .rodata : { *(.rodata*) } // 只读数据
    .data : { *(.data*) }    // 已初始化数据
    .bss : { *(.bss) *(COMMON) } // 未初始化数据
}
```

为什么裸机开发要手动写链接脚本？因为不像 Linux 应用有操作系统帮你分配地址，裸机程序得自己决定代码放到内存的哪里、栈在哪、数据在哪。

#### 3. objcopy — 格式转换器

干的活：从 `.elf` 中提取纯二进制数据，去掉 ELF 头和调试信息，生成可以直接烧录的 `.bin`。

```bash
arm-linux-gnueabihf-objcopy -O binary -S led.elf led.bin
```

- `-O binary`：输出格式为 raw binary
- `-S`：去掉符号表和重定位信息

ELF 文件包含大量元信息（段表、符号表、调试信息），芯片不认识这些，只认纯机器码。objcopy 就是把机器码部分抠出来。

#### 4. objdump — 反汇编器

干的活：把 `.elf` 反汇编成人类可读的汇编代码，用于调试和分析。

```bash
arm-linux-gnueabihf-objdump -D led.elf > led.dis
```

- `-D`：反汇编所有段
- 生成的 `.dis` 文件左边是机器码，右边是汇编指令，一一对应

用途：检查链接地址对不对、确认指令编码、排查跑飞问题。

### 速查表

| 工具 | 作用 | 输入 → 输出 | 典型命令 |
|------|------|-------------|----------|
| gcc | 编译器 | .c/.S → .o | `arm-linux-gnueabihf-gcc -c led.S -o led.o` |
| ld | 链接器 | .o → .elf | `arm-linux-gnueabihf-ld -Txxx.lds *.o -o xxx.elf` |
| objcopy | 格式转换 | .elf → .bin | `arm-linux-gnueabihf-objcopy -O binary -S xxx.elf xxx.bin` |
| objdump | 反汇编 | .elf → .dis | `arm-linux-gnueabihf-objdump -D xxx.elf > xxx.dis` |

### Makefile 中的典型写法

实际项目中用 Makefile 把这四步串起来：

```makefile
CROSS = arm-linux-gnueabihf-
CC = $(CROSS)gcc
LD = $(CROSS)ld
OBJCOPY = $(CROSS)objcopy
OBJDUMP = $(CROSS)objdump

TARGET = led
OBJS = start.o main.o

$(TARGET).bin: $(OBJS)
    $(LD) -Timx6ull.lds $(OBJS) -o $(TARGET).elf
    $(OBJCOPY) -O binary -S $(TARGET).elf $(TARGET).bin
    $(OBJDUMP) -D $(TARGET).elf > $(TARGET).dis

%.o: %.S
    $(CC) -c $< -o $@

%.o: %.c
    $(CC) -c $< -o $@
```

最终产物三个文件：
- `led.elf` — 完整可执行文件，带调试信息，用于 objdump 分析
- `led.bin` — 纯二进制，烧到板子里的就是这个
- `led.dis` — 反汇编文件，调试看这个

## 二、内存段：.text / .data / .bss / COMMON / .rodata

### 为什么分这么多段

程序运行时不同数据有不同的特性：有些只能读不能写、有些需要初始值、有些不需要初始值。编译器根据这些特性把数据分到不同的段里，链接器再按段的属性放到内存的不同区域。

### 五大段详解

#### .text — 代码段

存放编译后的机器指令（函数代码）。

- 只读（运行时不修改）
- 在 Flash/ROM 中可以直接执行（XIP）
- 链接脚本：`.text : { *(.text*) }`

```
void main(void) { ... }   // 函数代码 → .text
```

#### .rodata — 只读数据段

存放程序中不可修改的常量数据：

| 数据类型 | 示例 |
|----------|------|
| 字符串常量 | `printf("hello")` 中的 "hello" |
| const 全局变量 | `const int MAX = 100;` |
| 全局只读数组 | `const int table[] = {1,2,3};` |
| switch 跳转表 | 编译器自动生成 |
| 浮点数常量 | `3.14` 等 |

- 只读，写入会触发异常
- 链接脚本：`.rodata : { *(.rodata*) }`

```c
const int MAX_VALUE = 100;       // → .rodata
char *str = "Hello World";       // "Hello World" → .rodata，str 指针本身 → .data
const int lookup[] = {1, 2, 3};  // → .rodata
```

注意：局部变量加 const 不一定进 .rodata（可能在栈上），全局 const 才确定进 .rodata。

#### .data — 已初始化数据段

存放**初始化为非零值**的全局变量和静态变量。

- 需要在程序文件中保存初始值（占用 Flash 空间）
- 运行时可读可写（在 RAM 中）
- 链接脚本：`.data : { *(.data*) }`

```c
int global_var = 100;              // → .data
int global_array[] = {1, 2, 3};    // → .data
static int count = 50;             // → .data
```

#### .bss — 未初始化数据段

存放**未初始化或初始化为 0** 的静态变量和全局变量。

- 不占用 Flash/ROM 空间（不需要存初始值，反正都是 0）
- 运行时在 RAM 中，启动代码负责清零
- 链接脚本：`.bss : { *(.bss) *(COMMON) }`

```c
static int bss_var;           // → .bss（未初始化的 static）
int explicit_var = 0;         // → .bss（显式初始化为 0）
```

为什么 .bss 不占 Flash？因为全是 0，没必要在二进制文件里存一堆 0。链接器只记录 .bss 段的大小和起始地址，启动时由代码把那块内存清零就行。这也是为什么 .bss 段变量不能假定有初始值——上电后必须先清零才能用。

#### COMMON — 公共段

存放**未初始化的非静态全局变量**。

- 和 .bss 类似（都是 0），但区别在于：COMMON 段的符号大小和位置在**链接时**才确定
- 允许多个目标文件定义同名符号（链接器合并取最大）
- 链接脚本中通常和 .bss 放一起：`.bss : { *(.bss) *(COMMON) }`

```c
int common_var;               // → COMMON（非 static 的未初始化全局变量）
```

COMMON 段是 C 语言"暂定定义"（tentative definition）的产物。多个 .c 文件里都写 `int x;` 不报错，就是靠 COMMON 段合并实现的。

### 五段对比表

| 段 | 初始化 | 运行时可写 | 占用 Flash | 位置确定时机 | 典型变量 |
|----|--------|-----------|-----------|-------------|----------|
| .text | 不需要（是指令） | 只读 | 是 | 编译时 | 函数体 |
| .rodata | 需要初始值 | 只读 | 是 | 编译时 | const 全局、字符串 |
| .data | 需要非零初始值 | 可写 | 是 | 编译时 | `int g = 100;` |
| .bss | 自动清零 | 可写 | 否 | 编译时 | `static int s;` |
| COMMON | 自动清零 | 可写 | 否 | 链接时 | `int g;`（非 static） |

### .bss vs COMMON 的区别

这两个最容易搞混：

| 对比项 | .bss | COMMON |
|--------|------|--------|
| 变量类型 | static 未初始化 / 初始化为0 | 非 static 未初始化全局变量 |
| 大小确定 | 编译时 | 链接时 |
| 同名符号 | 不允许重复 | 允许多个目标文件同名合并 |
| 链接脚本 | `*(.bss)` | `*(COMMON)` |

一句话总结：加 static 的进 .bss，不加的进 COMMON。链接时 COMMON 会被合并进 .bss 段的内存区域。

### 链接脚本中的通配符

```
SECTIONS
{
    .text :     { *(.text*) }
    .rodata :   { *(.rodata*) }
    .data :     { *(.data*) }
    .bss :      { *(.bss) *(COMMON) }
}
```

`*(.rodata*)` 里两个 `*` 含义不同：
- 第一个 `*`：收集所有输入文件中的该段
- 第二个 `*`：匹配 `.rodata` 开头的所有段名（如 `.rodata.str1.1`、`.rodata.cst4` 等）

如果只写 `*(.rodata)` 不加第二个 `*`，字符串常量等数据会漏掉，程序可能跑飞。

### 内存布局示意（i.MX6ULL DDR 0x80000000 起）

```
高地址
  ┌──────────────┐
  │    栈区       │ ← SP 初始位置，向下生长（满减栈）
  ├──────────────┤
  │    堆区       │ ← malloc 用，向上生长
  ├──────────────┤
  │   .bss/COMMON │ ← 未初始化数据，启动时清零
  ├──────────────┤
  │    .data      │ ← 已初始化数据，从 Flash 拷贝到 RAM
  ├──────────────┤
  │   .rodata     │ ← 只读数据
  ├──────────────┤
  │    .text      │ ← 代码段
  └──────────────┘ ← 链接起始地址 0x87800000
低地址
```

### 易错

1. **`int x = 0;` 进 .bss 不进 .data**。编译器优化：初始化为 0 等价于不初始化，省 Flash 空间
2. **`const` 局部变量不一定进 .rodata**。可能优化到寄存器或栈上，只有全局 const 才确定进 .rodata
3. **字符串常量在 .rodata，但 `char *str` 指针本身在 .data**。别搞混常量和指向常量的指针
4. **.bss 不占 Flash 但占 RAM**。声明大数组 `int buf[10000];` 即使不初始化也会吃 40KB RAM
5. **COMMON 段是 C 的"暂定定义"特性**。`int x;` 写在多个 .c 文件里不报错靠的就是它，但这是隐患，应该用 `extern` 声明 + 一个文件定义
